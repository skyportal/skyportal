# Incremental TypeScript Migration Plan

This document describes how SkyPortal's frontend (`static/js`) moves to TypeScript
**file by file, over ~a year of normal frontend work**, with no big-bang rewrite
and no freeze on feature development. It is written against the actual state of
the codebase as of this branch.

## Goals

- Catch a class of bugs the Selenium suite cannot: wrong prop shapes, missing
  fields on API payloads, `undefined`/`null` access, incorrect Redux state
  access, mismatched action payloads.
- Replace ad-hoc `PropTypes` with compile-time types that also document the
  data model (sources, photometry, candidates, GCN events, allocations).
- Keep `allowJs` on the whole time so `.js`/`.jsx` and `.ts`/`.tsx` coexist; a
  PR migrates a handful of files and is reviewable on its own.

## Non-goals

- Not a rewrite. Behavior must not change in a migration-only PR.
- Not coupled to the Vite swap (see `doc/vite_migration.md`) — TypeScript works
  under the current rspack build too. Do them independently.
- Not a hooks-conversion effort. That is nearly done already (see below); do not
  bundle unrelated refactors into type-only PRs.

## Current state (measured, not assumed)

- **425 source files**: 323 `.jsx`, 102 `.js`, plus 5 `*.jsx.template` (the
  templated entry points, generated by `baselayer/tools/fill_conf_values.py`).
- **Only 2 class components remain**: `static/js/components/ErrorBoundary.jsx`
  (legitimately a class — `componentDidCatch`; either keep it or adopt
  `react-error-boundary`) and `static/js/components/FoldBox.jsx` (convertible).
  The "unify the state model on hooks" goal is effectively already done; finish
  it as a one-off, not as part of the type migration.
- **Build**: rspack (`rspack.config.js`) via `babel-loader` with
  `@babel/preset-env` + `@babel/preset-react`, restricted to `static/js`. The
  actual bundle command lives in the `baselayer` submodule
  (`baselayer/Makefile`: `bundle: npx rspack`).
- **Lint**: ESLint flat config v9 (`eslint.config.js`) with `@babel/eslint-parser`.
- **State**: Redux + `redux-thunk`, many `ducks/*.js` modules and a `store.js`.
- **UI**: MUI v7, `@rjsf` v6, `@mui/x-data-grid` v8 (ships its own types).

## Phase 0 — Tooling (one PR, no `.ts` files yet)

1. **Add dev deps**: `typescript`, `@babel/preset-typescript`,
   `typescript-eslint`, and the `@types/*` for libraries lacking bundled types
   (most MUI/x packages ship their own; check `react-redux`, `react-plotly.js`,
   `d3-*`, `mui-datatables`, `dygraphs`).
2. **`tsconfig.json`** at repo root — permissive to start:
   ```jsonc
   {
     "compilerOptions": {
       "target": "ES2020",
       "lib": ["ES2021", "DOM", "DOM.Iterable"],
       "jsx": "react-jsx",
       "module": "ESNext",
       "moduleResolution": "Bundler",
       "allowJs": true,           // .js/.jsx keep compiling
       "checkJs": false,          // don't type-check JS yet
       "noEmit": true,            // tsc is a checker only; rspack/babel emit
       "strict": false,           // ratcheted up later (Phase 3)
       "skipLibCheck": true,
       "esModuleInterop": true,
       "forceConsistentCasingInFileNames": true,
       "baseUrl": ".",
       "paths": { "baselayer/*": ["baselayer/static/js/*"] }
     },
     "include": ["static/js"]
   }
   ```
3. **Teach the build about `.ts`/`.tsx`** (rspack):
   - Extend the loader test to `/\.(js|jsx|ts|tsx)?$/` and add
     `@babel/preset-typescript` to the presets.
   - Add `.ts`, `.tsx` to `resolve.extensions`.
   - Babel only *strips* types (no checking) — that is intentional. Type-checking
     is a separate `tsc --noEmit` step so a type error fails CI without blocking
     local bundling.
4. **Lint**: add a `typescript-eslint` block in `eslint.config.js` scoped to
   `**/*.{ts,tsx}` using `@typescript-eslint/parser`; keep the babel parser for
   `.js`/`.jsx`.
5. **CI**: add `tsc --noEmit` to the frontend lint job. With zero `.ts` files and
   `checkJs:false` this is instantly green — it becomes the ratchet for later
   phases.

Phase 0 changes no runtime code and produces an identical bundle. Verify with
`npx rspack build` (the only thing that catches module-resolution regressions —
lint does not) before and after.

## Phase 1 — The type foundation (`static/js/types/`)

Before converting components, model the data they share. Create `static/js/types/`
with hand-written interfaces for the core domain objects, derived from the API
responses and the existing `PropTypes`:

- `Source`, `Photometry`/`PhotometryPoint`, `Candidate`, `Annotation`,
  `Classification`, `Group`, `GcnEvent`, `Allocation`, `Instrument`,
  `Telescope`, `Spectrum`, `User`/`Profile`.
- A `RootState` type for the Redux store, assembled from per-duck state types.
- A typed `dispatch`/`thunk` helper and typed `useSelector`/`useDispatch`
  (`useAppSelector`, `useAppDispatch`).

These are the highest-leverage types: every component consuming a `source` or
`photometry` benefits immediately. Keep them additive — JS files can import a
`.ts` type-only module with no runtime change.

## Phase 2 — Leaf-up migration order

Migrate in dependency order so a converted file imports already-typed modules:

1. **Pure utilities** — `units.js`, `wavelengthConverter.js`,
   `utils/*`, `instrumentTrie.js`, `usernameTrie.js`. No React; easiest wins;
   surfaces the most "this was secretly `any`" bugs.
2. **API + store layer** — `API.js`, `store.js`, then `ducks/*.js`. Type the
   action creators, thunks, and reducer state against the Phase-1 types. This is
   where wrong-payload bugs hide.
3. **Leaf components** — small presentational components with few props
   (`Button.jsx`, chips, badges, dialogs). Convert `propTypes` → a `Props`
   interface; delete the `PropTypes` block in the same PR.
4. **Mid-tier components** — tables, forms, plots (the just-migrated
   `SourceTable`, `PhotometryTable`, `StyledDataGrid`, `PhotometryPlot`). These
   benefit most from `GridColDef<Source>` and typed Plotly/Vega specs.
5. **Pages / templates** — top-level route components last, once their children
   are typed. Note the `*.jsx.template` entry points: rename to `*.tsx.template`
   and confirm `fill_conf_values.py` still matches them (it globs `*.template`).

Rule of thumb per PR: **5–15 files, one subsystem, green `tsc --noEmit`, green
`rspack build`, no behavior change.** A renamed `.jsx`→`.tsx` plus its `Props`
interface and the removal of its `PropTypes` is the canonical unit.

## Phase 3 — Strictness ratchet

Start at `strict: false`; tighten once the bulk is converted:

1. `noImplicitAny` → fix the explicit `any`s that were hidden.
2. `strictNullChecks` → the biggest bug-catcher; expect the most churn here
   (optional API fields, `state.x?.y`).
3. `strict: true` once null-checks are clean.
4. Optionally flip `checkJs: true` to type-check the remaining `.js` via JSDoc
   as a finish line, then delete it when the last `.js` is gone.

Enforce monotonic progress with a CI check that the count of `.js`/`.jsx` files
only ever decreases (a simple `find | wc -l` compared against a committed
baseline), so the migration cannot silently stall or regress.

## Conventions

- **Props**: replace each `Component.propTypes` with an exported `Props`
  interface and `({ ... }: Props)`. Remove `defaultProps` in favor of default
  parameter values (React is deprecating `defaultProps` for function comps).
- **Redux**: use the typed `useAppSelector`/`useAppDispatch`; never annotate
  `state` as `any`. Thunks return `AppThunk<ReturnType>`.
- **MUI X**: type columns as `GridColDef<Source>[]`; `renderCell` params are
  `GridRenderCellParams<Source>`.
- **Don't** introduce `// @ts-ignore` to land a file — use `// @ts-expect-error`
  with a comment so it fails loudly when the underlying issue is fixed.

## Risks / gotchas specific to this repo

- The bundle command is in the **baselayer submodule**; Phase 0's loader change
  is in `rspack.config.js` (this repo), but if baselayer is later swapped to
  Vite, re-confirm TS handling there (esbuild strips types; still run
  `tsc --noEmit` separately).
- Some libraries are untyped or stale (`mui-datatables` is being retired in
  favor of `@mui/x-data-grid`; `dygraphs`, `underscore.template`, `kapsule`).
  Add minimal `static/js/types/<lib>.d.ts` shims rather than blocking on
  upstream types.
- `@rjsf` v6 generics are heavy; type form schemas as `RJSFSchema` and avoid
  over-typing `formData` early.
- Keep type-only PRs free of logic changes so reviewers can trust "no behavior
  change" and the Selenium suite stays a meaningful safety net.
